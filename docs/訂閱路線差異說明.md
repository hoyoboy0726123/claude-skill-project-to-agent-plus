> 所有結論皆經實戰驗證(SRT AV Studio + textkit E2E 測試,2026-07)。

## 一、一張圖看懂本質差異

```
API-Key 路線(Gemini / Groq / OpenAI / Anthropic / Ollama)
  你的程式自己跑推理迴圈 ──API──> 模型回 tool_calls
  工具 = 同行程的 Python 函式直接呼叫          ← 全程透明、你全權掌控

訂閱 CLI 路線(Claude Code / codex)
  你的程式 spawn 官方 CLI ──它自己就是完整 agent──> 只還你最終回覆
  工具 = 獨立行程的 MCP server(stdio)         ← 迴圈黑盒、工具跨行程
```

**為什麼被迫不同**:訂閱額度(Pro/Max、ChatGPT 方案)只綁官方 client、不給 API key
→ 你不能自己跑迴圈 → 官方 CLI 唯一的外掛工具入口就是 MCP。
對模型而言工具呼叫機制其實一樣(都是 native tool use),差別全在「水管接誰家」。

## 二、選擇前必知的七條取捨

| # | 取捨 | 說明 |
|---|---|---|
| 1 | 🧠 **幻覺偵測失效** | 原 skill 逐輪掃 tool_calls 抓「嘴上說做了、實際沒做」;CLI 是黑盒看不到 → 改用「交付契約(回覆必附完整路徑)+ 系統偵測網(快照輸出目錄自動補漏)」雙保險 |
| 2 | 📁 **檔案自動傳送改道** | 原本 adapter 看 tool result 的 `output_file` key 自動送檔;CLI 路線看不到 tool result → 改掃「最終回覆文字」中的路徑,AI 漏報時偵測網補(偶爾晚一步) |
| 3 | 🔒 **codex 的 ACL 破口** | codex headless 必開 bypass 旗標 → 它自帶的原生 shell/檔案能力**不受你的資料夾白名單約束**。你的 ACL 只管得住「你的工具」。風險等級 ≈ skill 的「Host 模式 shell」 |
| 4 | 🧪 **嚴格沙盒只有一條路** | claude 安全模式(allowlist 只開 `mcp__*`)+ 你自己的 Docker 化 shell 工具 = 沙盒完整;**codex 路線做不到嚴格沙盒**(原生 shell 關不掉) |
| 5 | 📊 **額度天花板** | 吃你的訂閱額度;用罄直接 turn.failed(整合層會顯示明確錯誤+重置時間)。重度自動化(排程每 5 分鐘打一輪)會快速耗盡 |
| 6 | 👥 **多人隔離要繞路** | per-user 記憶隔離的 ContextVar 跨不過行程(工具活在 CLI spawn 的 MCP server 裡)→ 改走環境變數繼承鏈。**單人使用完全不受影響** |
| 7 | ✅ **不變的東西** | two-step 確認(預覽→確認→寫)、資料夾權限檢查、排程提醒——全部照舊,因為它們活在工具函式體內,誰來呼叫都一樣擋 |

**一個意外的好處**:self-evolution(Phase 12)反而更簡單——CLI 每輪 spawn 全新
MCP server,新工具下一輪自動生效,原本的 hot-reload 全套地雷(importlib cache、
schemas 不准 cache…)直接免除。

## 四、怎麼選(決策建議)

| 你的情況 | 建議 |
|---|---|
| 已付 Claude Pro/Max,想省 API 費 | **claude 安全模式** — 安全邊界最嚴、對話品質最好,推薦預設 |
| 已付 ChatGPT,想省 API 費 | codex — 接受取捨 #3/#4,適合個人環境;生圖是它的獨特優勢(image_gen 走訂閱) |
| 多人共用的 TG bot | API 路線較穩(per-user 隔離原生支援);訂閱路線要處理取捨 #6 且額度共搶 |
| 需要嚴格沙盒(不信任 LLM) | claude 安全模式 + Docker 化 MCP shell;**不要選 codex** |
| 高頻自動化(排程/批次) | API 路線(pay-as-you-go 可控);訂閱額度會撞牆 |
| 完全不想花錢 | 原路線的 Gemini 免費 tier / Ollama 本地 |

兩條路**不互斥**:平常用訂閱聊天省錢,批次任務切 API,settings 一鍵換。
